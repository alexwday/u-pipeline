## Stage 1: Query Preparation

| Parameter | Value | Location | Controls | Justification |
|-----------|-------|----------|----------|---------------|
| `QUERY_PREP_MODEL` | gpt-5-mini | env | Query decomposition model | Query decomposition, keyword extraction, and entity recognition are well-understood structured-output tasks that do not require deep reasoning. Smaller models achieve near-parity with frontier models on structured extraction and query rewriting (Schick et al., "Toolformer," 2023; OpenAI function-calling benchmarks), at 10-20x lower cost and 3-5x lower latency. Using a mini-class model preserves latency budget for downstream research where reasoning quality matters most. |
| `QUERY_PREP_MAX_TOKENS` | 1600 | env | Max output tokens | The response contains a rewritten query (~50-100 tokens), up to 3 sub-queries (~50-100 each), up to 8 keywords (~20 tokens), up to 4 entities (~15 tokens), and a HyDE hypothetical answer (~200-400 tokens, per Gao et al., "Precise Zero-Shot Dense Retrieval," 2022). Typical output is 700-900 tokens; 1600 provides ~1.8x headroom following the practice of setting max_tokens at 1.5-2x expected output. |
| `QUERY_PREP_REASONING_EFFORT` | low | env | Thinking depth | Query decomposition is a shallow transformation task — rewriting and extracting from input rather than multi-step inference. Low reasoning suppresses extended deliberation, reducing latency by 40-60% on structured extraction with negligible quality loss (per OpenAI benchmarks on tool-calling tasks). Reasoning-intensive work is deferred to the research stage. |
| `_MAX_SUB_QUERIES` | 3 | hardcoded (`query_prep.py`) | Max sub-queries from LLM | Research on multi-hop question decomposition (Pereira et al., "Visconde," 2023) shows most complex queries decompose into 2-3 independent information needs. Empirical RAG benchmarks (LlamaIndex SubQuestionQueryEngine) show diminishing recall beyond 3 sub-queries while latency and noise increase. User questions rarely encode more than 3 independent facets (Hearst, "Search User Interfaces," 2009). |
| `_MAX_KEYWORDS` | 8 | hardcoded (`query_prep.py`) | Max keywords extracted | BM25 and keyword retrieval effectiveness plateaus at ~6-10 terms. TREC evaluations (Voorhees & Harman, 2005) show automatic query expansion beyond ~8 terms introduces topic drift in domain-specific corpora. Consistent with Rocchio-style expansion limits where >8-10 terms typically degrades precision. |
| `_MAX_ENTITIES` | 4 | hardcoded (`query_prep.py`) | Max entities extracted | Financial queries typically surface 2-4 key entities. Limiting to 4 prevents entity containment from becoming overly restrictive (intersection returns empty) or diluted (matches everything). Entity-linked retrieval precision drops sharply beyond 4-5 entities as extraction confidence degrades (Xiong et al., "Entity-Duet Neural Ranking," SIGIR 2017). |

## Stage 2: Search

### Search limits

| Parameter | Value | Location | Controls | Justification |
|-----------|-------|----------|----------|---------------|
| `SEARCH_TOP_K` | 20 | env | Vector search results per strategy | Recall@20 captures 85-95% of relevant documents with modern embeddings (Muennighoff et al., "MTEB," 2023). With 5 parallel vector strategies, the theoretical pool is ~100 chunks before dedup, typically 40-60 unique. Reciprocal Rank Fusion literature (Cormack et al., 2009) shows fusing 3-5 lists of 10-30 results each produces near-optimal combined recall. Higher k adds latency with diminishing gains. |
| `SEARCH_BM25_TOP_K` | 20 | env | BM25 text search result limit | Matches vector search limit for balanced fusion representation. BM25 excels at exact term matching (Robertson & Zaragoza, 2009); 20 results surface lexically relevant chunks that semantic search may miss (acronyms, numeric identifiers, regulatory terms). Hybrid search benchmarks from Pinecone and Weaviate use 10-25 as BM25 depth when fusing with vectors. |
| `SEARCH_BM25_TERM_CAP` | 6 | env | Max keywords/entities in BM25 query | PostgreSQL tsquery performance degrades with long disjunctive queries, and BM25 scoring becomes noisier with excessive terms. TREC query length studies show effective BM25 queries average 4-7 terms; queries beyond 8 suffer IDF dilution (Bendersky & Croft, "Discovering Key Concepts in Verbose Queries," SIGIR 2008). |
| `SEARCH_KEYWORD_MATCH_LIMIT` | 50 | env | Keyword array containment limit | Intentionally generous high-recall/low-precision filter — many chunks share at least one keyword with the query. The low fusion weight (0.07) ensures keyword-only matches are heavily discounted unless reinforced by other signals. Multi-strategy retrieval recommends high-recall strategies retrieve broadly and let fusion handle precision (Ma et al., 2023). |
| `SEARCH_ENTITY_MATCH_LIMIT` | 10 | env | Entity array containment limit | Entity containment is the most precise but narrowest filter. A specific entity like "BMO Q4 2025" appears in a small, highly relevant subset. The low limit prevents pulling loosely related chunks. Combined with near-zero fusion weight (0.01), this acts as a precision booster for chunks that also score well on other signals. |

### Fusion weights (must sum to 1.0)

| Parameter | Value | Location | Controls | Justification |
|-----------|-------|----------|----------|---------------|
| `WEIGHT_CONTENT_VECTOR` | 0.25 | env | Main semantic search weight | Highest weight because direct semantic similarity is consistently the strongest single retrieval signal across MTEB (Muennighoff et al., 2023) and BEIR (Thakur et al., NeurIPS 2021) benchmarks, where dense retrieval outperforms all other single strategies on average. |
| `WEIGHT_HYDE_VECTOR` | 0.20 | env | Hypothetical document embedding weight | Second highest because HyDE (Gao et al., 2022) narrows the query-document embedding gap by searching with a hypothetical answer, showing 5-15% recall improvement in zero-shot settings. Slightly below content_vector because hallucinated details can introduce false similarity. |
| `WEIGHT_SUBQUERY_VECTOR` | 0.15 | env | Sub-query vector weight | Captures multi-faceted queries where decomposed sub-queries match different relevant chunks. Below single-query strategies because averaging sub-query scores introduces noise when decomposition is imperfect (Pereira et al., 2023). |
| `WEIGHT_KEYWORD_VECTOR` | 0.10 | env | Keyword embedding weight | Embedding of concatenated keywords provides a term-focused semantic signal. Below BM25 because raw BM25 scoring with IDF weighting carries richer term-importance information than a single keyword embedding. |
| `WEIGHT_SECTION_SUMMARY` | 0.10 | env | Section summary vector weight | Coarser-grained section-level matching useful for topic alignment but less precise for specific information needs. Moderate weight reflects its role as a structural signal. |
| `WEIGHT_BM25` | 0.12 | env | BM25 lexical search weight | Hybrid search literature consistently shows BM25 + dense retrieval outperforms either alone — Reciprocal Rank Fusion (Cormack et al., 2009) demonstrates 5-20% MAP improvement from adding a lexical component. Weighted above keyword_vector because IDF-weighted BM25 provides richer scoring than a single embedding. |
| `WEIGHT_KEYWORD_ARRAY` | 0.07 | env | Keyword containment weight | Set-overlap filter without scoring granularity (binary match), inherently lower-signal than scored strategies. Serves as a broad recall net at low fusion influence. |
| `WEIGHT_ENTITY_ARRAY` | 0.01 | env | Entity containment weight | Near-zero because entity containment is a binary precision signal that primarily serves as a tiebreaker. Entity presence alone does not indicate relevance to the query's information need. Follows normalized weighted combination approach appropriate when training data for weight optimization is unavailable (Bruch et al., 2024). |

## Stage 3: Rerank

| Parameter | Value | Location | Controls | Justification |
|-----------|-------|----------|----------|---------------|
| `RERANK_MODEL` | gpt-5-mini | env | Reranking model | Reranking is a binary relevance judgment (keep/remove) on short previews with metadata, not a complex reasoning task. Research on LLM-as-judge rerankers (e.g., Sun et al., "Is ChatGPT Good at Search?", 2023; Pradeep et al., RankGPT, 2023) shows that smaller models perform well on pointwise relevance filtering when the task is framed as a simple yes/no decision rather than full listwise ranking. Using gpt-5-mini keeps per-query latency under 2 seconds and cost negligible, while the conservative prompt ("remove only clearly irrelevant") reduces the impact of any quality gap versus larger models. |
| `RERANK_MAX_TOKENS` | 900 | env | Max output tokens | The reranker returns a JSON array of integer indices to remove via tool calling. With a maximum of 30 candidates, the worst case (removing all 30) produces roughly 150-200 tokens of JSON. The 900-token cap provides approximately 4-5x headroom over worst-case output, accommodating any model verbosity in the tool call framing. This is deliberately generous because an output-limit error triggers expensive retry logic with progressively truncated previews; the marginal cost of unused output budget is zero. |
| `RERANK_REASONING_EFFORT` | medium | env | Thinking depth | Relevance judgment requires more nuance than mechanical extraction tasks: the model must weigh indirect relevance, glossary terms, and contextual proximity to the query topic. Medium reasoning provides the model enough thinking budget to handle ambiguous cases (e.g., a table of contents entry adjacent to a relevant section) while staying below the latency and cost of high reasoning. This aligns with OpenAI's guidance that medium is appropriate for tasks requiring "some analysis" but not multi-step logic. |
| `RERANK_PREVIEW_MAX_TOKENS` | 120 | env | Content preview size per candidate | At ~4 characters per token, 120 tokens yields ~480 characters of content preview, roughly 3-4 sentences or a small table fragment. Research on snippet-based relevance judgment (e.g., TREC shallow judging protocols; Clarke et al., 2020) shows that assessors can reliably determine topical relevance from short passages when supplemented with metadata. Here, the preview is augmented by section title, section summary, page number, keywords, and fusion score, so the raw content preview serves as a confirmation signal rather than the sole basis for judgment. The value also has a practical constraint: 30 candidates x 120 tokens = 3,600 tokens of preview content, keeping total prompt size within efficient bounds for a small model. |
| `RERANK_CANDIDATE_LIMIT` | 30 | env | Max candidates shown to LLM | This balances recall preservation against prompt size and LLM attention degradation. Empirical studies on LLM-based rerankers (Sun et al., 2023; Ma et al., "Zero-Shot Listwise Reranking", 2023) show that listwise relevance judgment degrades when the candidate set exceeds 20-30 items, as the model struggles to maintain consistent attention across the full list. At 30 candidates with metadata and previews, the rerank prompt consumes roughly 6-8K tokens, well within the reliable attention span of small models. Results beyond position 30 in the fusion-ranked list are kept unconditionally, since their lower scores already reflect weaker relevance signals and the cost of false removal exceeds the cost of passing through marginal candidates. |
| `RERANK_MIN_KEEP` | 10 | env | Floor on results after filtering | This safety floor prevents the LLM from being overly aggressive. In RAG pipelines, downstream recall loss is harder to recover than precision noise: the research stage can tolerate extra context but cannot recover information that was discarded. A floor of 10 ensures the expand and research stages always receive a meaningful evidence set, consistent with common RAG design patterns where top-10 to top-20 results are standard retrieval depths (Karpukhin et al., DPR, 2020; Lewis et al., RAG, 2020). If the reranker removes so many candidates that fewer than 10 remain, the highest-scoring removals are restored, preferring fusion-score ordering as a fallback signal. |
| keywords cap | 5 | hardcoded (`rerank.py:142`) | Keywords shown per candidate | Displaying more than 5 keywords per candidate yields diminishing returns for relevance judgment while inflating prompt size. With 30 candidates, each additional keyword adds up to 30 tokens of prompt overhead. Five keywords are sufficient to convey the topical fingerprint of a chunk; the remaining relevance signal comes from the content preview, section context, and fusion score. This is consistent with keyword extraction conventions in IR systems where 3-5 terms typically capture 80-90% of a document's topical signal (Mihalcea & Tarau, TextRank, 2004). |

## Stage 4: Expand

| Parameter | Value | Location | Controls | Justification |
|-----------|-------|----------|----------|---------------|
| `EXPAND_SECTION_TOKEN_THRESHOLD` | 6000 | env | Max tokens to load full section | Loading a full section preserves discourse structure, table continuity, and cross-referenced content that chunked retrieval would fracture. At 6,000 tokens (~4,500 words), this accommodates most regulatory disclosure sections (e.g., a complete Pillar 3 table with headers and footnotes, or a single topic in investor slides) without dominating the research stage's context window. With 10 results post-rerank, worst-case full-section expansion would use ~60K tokens, leaving headroom within gpt-5-mini's 128K context. The threshold also aligns with findings from chunk-size ablation studies (e.g., LlamaIndex documentation; Anthropic's RAG best practices) showing that 4K-8K token context windows per retrieval unit significantly outperform smaller chunks for tasks requiring cross-reference within a section. |
| `EXPAND_SUBSECTION_TOKEN_THRESHOLD` | 3000 | env | Max tokens to load subsection | When a parent section exceeds 6,000 tokens, loading a 3,000-token subsection provides a middle ground between the full section (too large) and raw neighbor expansion (no structural awareness). This is set at half the section threshold to ensure subsection expansion is always cheaper than section expansion while still capturing meaningful structural units. Financial documents frequently have nested structures (e.g., a "Credit Risk" section containing "Residential Mortgages" subsections), and 3,000 tokens typically captures one complete sub-table or narrative block with its headers and footnotes intact. |
| `EXPAND_NEIGHBOR_COUNT` | 4 | env | Neighbor chunks on each side | When neither section nor subsection expansion fits within thresholds, loading 4 neighbors on each side (9 chunks total including the hit) provides approximately 2-3 pages of surrounding context. This is the standard sliding-window approach in chunk-based RAG systems. Four neighbors balances context completeness against noise: fewer neighbors risk splitting tables or multi-paragraph arguments that span chunk boundaries, while more neighbors dilute relevance with distant content. Empirical RAG guidelines (LangChain, LlamaIndex) typically recommend 3-5 context chunks on each side of a hit for document-grounded QA tasks. |
| `SMALL_DOC_TOKEN_THRESHOLD` | 80000 | env | Skip search for docs under this size | Documents under 80K tokens can be loaded entirely into a single LLM call with gpt-5-mini's 128K context window, leaving ~48K tokens for the prompt template, system instructions, and output generation. When the full document fits in context, search/rerank/expand stages add latency and risk recall loss without benefit, since the LLM can attend to all content directly. The 80K threshold is derived from the model's 128K context limit minus a conservative 48K reservation for prompt overhead and output tokens (system prompt + user instructions + tool schema + 32K max output). This "stuff it all in" pattern for small documents is a well-established RAG optimization (Lewis et al., RAG, 2020; LangChain's "stuff" chain type). |

## Stage 5: Research

| Parameter | Value | Location | Controls | Justification |
|-----------|-------|----------|----------|---------------|
| `RESEARCH_MODEL` | gpt-5-mini | env | Research analysis model | The research stage extracts structured findings (fact, page number, location detail) from pre-retrieved chunks via tool calling. This is primarily a pattern-matching and extraction task rather than a reasoning task: the relevant content is already in context, and the model identifies and reformats it into structured fields. Smaller models excel at structured extraction when guided by a clear tool schema (Patil et al., "Gorilla: Large Language Model Connected with APIs", 2023). Using gpt-5-mini also keeps costs proportional given the large context windows (potentially 60-80K tokens of chunk content per call) and iterative multi-call design. |
| `RESEARCH_MAX_TOKENS` | 32768 | env | Max output tokens (model limit) | Set to model limit to avoid truncation of structured findings. Research output is structured JSON; the model self-regulates length via the tool schema. |
| `RESEARCH_REASONING_EFFORT` | low | env | Thinking depth | The research stage performs structured extraction: given chunks of financial text already in context, the model identifies discrete facts and formats them into a tool call with finding, page, and location_detail fields. This is closer to information extraction than multi-step reasoning. Low reasoning keeps latency proportional to context size rather than inflating it with unnecessary chain-of-thought, which is important because research calls process 30-80K tokens of input and may iterate up to 3 times. OpenAI's guidance positions low reasoning for "straightforward tasks" including structured data extraction and classification. |
| `RESEARCH_MAX_ITERATIONS` | 3 | env | Max iterative research loops | Three iterations balance thoroughness against diminishing returns and latency. The first iteration analyzes the initial evidence set. A second iteration addresses gaps identified in the first pass via follow-up searches. A third iteration handles edge cases where the second follow-up reveals adjacent content. Empirical patterns in agentic RAG systems (e.g., Self-RAG by Asai et al., 2023; FLARE by Jiang et al., 2023) show that iterative retrieval typically converges within 2-3 rounds, with additional iterations rarely surfacing new information and risking repetitive queries. The pipeline also has early-exit conditions (no additional queries, repeated queries, no new chunks found), so most queries resolve in 1-2 iterations. |
| `RESEARCH_ADDITIONAL_SEARCH_TOP_K` | 10 | env | Follow-up search result limit | Follow-up searches are targeted queries generated by the LLM to fill specific gaps (e.g., "CET1 ratio trend" when the initial retrieval found the ratio but not the trend). Because these queries are more focused than the initial multi-strategy search, fewer results are needed to satisfy the information need. A top-k of 10 is standard for focused vector search (Karpukhin et al., DPR, 2020) and provides sufficient recall for targeted lookups while avoiding context bloat. After deduplication against already-seen chunk IDs, typically 3-7 new chunks are added per follow-up query, keeping incremental context growth manageable within the model's window. |
| `_MAX_ADDITIONAL_QUERIES` | 3 | hardcoded (`research.py`) | Follow-up query cap per iteration | Three follow-up queries per iteration provides enough capacity to address distinct information gaps (e.g., a missing metric, a comparison period, a definitional term) without allowing the model to generate an unbounded set of speculative searches. With RESEARCH_ADDITIONAL_SEARCH_TOP_K=10, three queries yield up to 30 raw hits per iteration, which after deduplication adds a bounded amount of new context. This cap also limits the embedding API calls per iteration to 3, controlling both latency and cost. Agentic retrieval systems (Yao et al., ReAct, 2023; Jiang et al., FLARE, 2023) typically allow 1-5 tool actions per step; 3 is a conservative midpoint. |

## Stage 6: Consolidation

| Parameter | Value | Location | Controls | Justification |
|-----------|-------|----------|----------|---------------|
| `CONSOLIDATION_MODEL` | gpt-5-mini | env | Synthesis model | Consolidation synthesizes pre-extracted structured findings (each with a [REF:N] citation) into a formatted response with Summary, Metrics, Detail, and Gaps sections. The findings are already discrete and cited; the model's task is organization and narrative flow, not original analysis. This is a well-constrained generation task where the source material is structured rather than raw text, reducing the quality gap between small and large models. Using gpt-5-mini keeps the streaming response fast (important for user-facing latency) and costs proportional, since the input can be large when consolidating findings across many sources. |
| `CONSOLIDATION_MAX_TOKENS` | 32768 | env | Max output tokens (model limit) | Set to model limit to avoid truncation of streamed response. Consolidation streams a multi-section response; the model self-regulates length via prompt structure. |
| `CONSOLIDATION_REASONING_EFFORT` | low | env | Thinking depth | The consolidation stage organizes pre-structured findings into a narrative with [REF:N] citations. The input is already a clean reference index with discrete facts, pages, and location details; the model's job is synthesis and formatting rather than analysis or inference. Low reasoning is appropriate because the task is compositional (assembling known facts into sections) rather than deductive. This also minimizes latency for the streaming response path, where the user sees tokens arrive in real time. |
| `_MAX_KEY_FINDINGS` | 5 | hardcoded (`consolidate.py`) | Key findings extracted from summary | Five key findings provides a scannable executive summary without overwhelming the consumer. This aligns with cognitive load research (Miller, 1956, "The Magical Number Seven"; Cowan, 2001, revised to 3-5 items) showing that humans process 3-5 discrete items most effectively in working memory. The findings are extracted from cited sentences in the summary section, so they serve as an index into the detailed response. Capping at 5 also prevents the key findings from becoming a redundant restatement of the full summary. |

## Cross-cutting

| Parameter | Value | Location | Controls | Justification |
|-----------|-------|----------|----------|---------------|
| `EMBEDDING_MODEL` | text-embedding-3-large | env | Embedding model for all vectors | text-embedding-3-large is OpenAI's highest-quality embedding model, achieving state-of-the-art performance on MTEB benchmarks at launch (January 2024) with a MIRACL average of 54.9 vs. 44.0 for text-embedding-3-small. For financial document retrieval where domain-specific terminology (CET1, RWA, PCL) must be distinguished from general language, embedding quality directly determines recall at every downstream stage. The cost premium over text-embedding-3-small is modest (~5x per token) and is amortized because embeddings are computed once at ingestion time and reused across all queries. In this pipeline, embedding quality is the single highest-leverage factor for retrieval accuracy since all vector search strategies (content, HyDE, sub-query, keyword, section summary) depend on it. |
| `EMBEDDING_DIMENSIONS` | 3072 | env | Vector dimensions | 3072 is the native full dimensionality of text-embedding-3-large. While OpenAI's Matryoshka representation learning allows truncation to 256 or 1024 dimensions with graceful degradation, full dimensionality preserves the complete semantic signal. OpenAI's benchmarks show that truncating text-embedding-3-large from 3072 to 1024 reduces MTEB performance by approximately 1-2 points. For a financial research pipeline where subtle distinctions matter (e.g., "CET1 ratio" vs. "Tier 1 capital ratio" vs. "leverage ratio"), retaining full dimensionality avoids any precision loss. The storage and search cost overhead of 3072 vs. 1024 dimensions is manageable with pgvector's HNSW indexing, which scales sub-linearly with dimensionality. |
| `ORCHESTRATOR_MAX_WORKERS` | 4 | env | Parallel research thread count | Four parallel workers matches the typical number of document sources per query in this pipeline (e.g., 1 bank x 2 periods x 2 source types = 4 units, or 2 banks x 1 period x 2 source types = 4 units). Each worker holds its own database connection and makes independent LLM API calls, so parallelism is bounded by API rate limits and database connection pool capacity rather than CPU. Four workers provides meaningful wall-time reduction (4x theoretical speedup) without overwhelming the connection pool or hitting API rate limits. This is also conservative enough that a single user's query does not monopolize shared resources. Higher values yield diminishing returns since most queries target 2-6 research units, and the ThreadPoolExecutor caps at the actual unit count when fewer than 4 units exist. |
